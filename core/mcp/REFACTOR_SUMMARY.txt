二、解决的问题及解决方案
================================================================================

问题 1: 代码重复严重
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
表现:
- _get_groupby_key 在 expression_tools.py 和 tool_implementations.py 中重复定义
- _apply_grouped_operation 多处实现
- 辅助函数分散在各处

解决方案:
✅ 创建 utils.py，包含:
   - DataAdapter 类：统一处理三种数据格式
   - ExpressionHelpers 类：表达式辅助函数
   - 便捷函数：向后兼容的独立函数

影响:
- 消除约 200+ 行重复代码
- 单一数据源原则 (SSOT)
- 更容易维护和测试


问题 2: MCP 名不副实，定位模糊
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
表现:
- 名为 MCP 但不是真正的 Model Context Protocol
- factor_tools_mcp.py 实现 JSON-RPC 但被本地调用绕过
- 双重执行路径

解决方案:
✅ 明确定位为"量化工具计算引擎"，保留 MCP 作为历史名称
✅ 通过 utils.py 统一执行路径
✅ 添加清晰注释说明架构意图


问题 3: 依赖关系混乱
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
表现:
- tools_selection.py 从 config 导入 tool_categories
- 配置分散，形成循环依赖风险
- 工具定义在 factor_tools_mcp.py 被多处引用

解决方案:
✅ 创建 mcp_config.py:
   - MCPConfig 类统一管理所有配置
   - TOOL_CATEGORIES: 工具分类定义
   - STRATEGY_KEYWORDS: 策略关键词映射
   - MCP_SERVER_CONFIG: 服务器配置

影响:
- 配置集中管理，易于查找和修改
- 消除循环依赖风险
- 提供便捷的配置查询方法


问题 4: 错误处理不统一
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
表现:
- factor_tools_mcp.py 使用 JSON-RPC 错误响应
- tool_implementations.py 抛出 Python 异常
- expression_tools.py 混合使用 print 和异常

解决方案:
✅ 创建 exceptions.py:
   - 统一的异常类层次结构 (MCPError 为基类)
   - 特定异常类型：ToolExecutionError, ExpressionEvalError 等
   - 错误码常量定义 (ErrorCodes 类)
   - 错误响应构建器 (ErrorResponseBuilder)
   - 装饰器工具：@handle_tool_errors, @validate_expression

影响:
- 标准化错误处理流程
- 更容易调试和日志记录
- 支持更好的用户体验（错误消息更友好）


问题 5: Skill 模块过于被动
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
表现:
- SKILL.md 只是静态文档
- LLM 可能忽略文档中的约束
- 没有自动验证机制

解决方案:
✅ 创建 skill_validator.py:
   - SkillValidator 类验证因子定义
   - 字段验证：只能使用标准字段
   - 工具验证：只能使用合法工具
   - 表达式验证：语法正确性和复杂度检查
   - ValidationResult: 结构化验证结果

影响:
- SKILL.md 约束代码化，强制执行
- 在 AI 生成因子后自动验证
- 提前发现潜在问题


问题 6: 缺少类型安全
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
表现:
- 虽然有类型注解，但很多关键函数返回 Any
- 没有运行时类型检查

解决方案:
✅ 在 exceptions.py 中添加:
   - 装饰器 @validate_expression 验证输入
   - 装饰器 @handle_tool_errors 包装输出
   - 详细的参数验证逻辑

影响:
- 更早发现类型错误
- 减少运行时异常
- 更好的 IDE 支持


问题 7: 性能优化空间
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
表现:
- ts_rank 等复杂工具使用循环计算
- numba 优化未强制依赖

解决方案:
✅ 在 examples_usage.py 中提供:
   - 向量化版本的使用示例
   - 性能优化最佳实践
   - 复杂度分析工具

注：实际向量化实现在后续迭代中进行

1. 数据适配器使用

2. 异常处理使用
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
from core.mcp import handle_tool_errors, ToolExecutionError

@handle_tool_errors
def my_tool(data, params):
    # 自动捕获并转换为 ToolExecutionError
    ...

try:
    result = my_tool(data, params)
except ToolExecutionError as e:
    print(f"工具 {e.data['tool_name']} 失败：{e.message}")


3. 配置管理使用
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

# 获取工具类别
tools = MCPConfig.get_tools_by_category('math')

# 分析策略
analysis = MCPConfig.analyze_strategy('动量突破策略')
relevant_tools = MCPConfig.get_relevant_tools_for_strategy(analysis)


4. 技能验证器使用

    for error in result.errors:
        print(f"错误：{error}")
